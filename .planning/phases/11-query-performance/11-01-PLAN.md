<objective>
Eliminate the paper-preflight N+1 query pattern so preflight issues a constant, small
number of SQL queries regardless of how many approved candidates exist.

Today `_build_paper_session_plan` (src/trading_platform/services/paper_execution.py:1184)
loops over every approved candidate and, per candidate, calls
`_resolve_paper_intent_decision` — which issues an exact-match `intent_hash` SELECT
(line 1252), a predecessor-lineage SELECT (line 1283), and sometimes a `session.get(PaperOrder)`
for reuse (line 1213). With M candidates that is ~2-3·M queries: a classic N+1.

Purpose: Preflight cost must be flat in portfolio size (PERF-01). This is a correctness-of-scale
requirement — the assertion must be a hard, testable invariant (query count is INVARIANT to
candidate count), not a "roughly constant" heuristic.
Output: A reusable query-counting test harness, a batched preflight that resolves all intent
decisions in-memory from one PaperOrder load, and an integration test that pins the invariant.
</objective>

def _build_paper_session_plan(
    session, *, strategy_id: str, as_of_session: date,
    requested_risk_run_id: str | None, failure_threshold: int,
    client_order_id_prefix: str,

    source_risk_run = _resolve_source_risk_run(session, strategy_id=..., as_of_session=..., requested_risk_run_id=...)  # 1 query (auto-resolve path)
    candidates = _load_submission_candidates(session, source_risk_run.id)  # 1 query (RiskEvent join Symbol)
    for candidate in candidates:                                           # <-- N+1 STARTS HERE
        intent_decision = _resolve_paper_intent_decision(session, ..., candidate=candidate, ...)
        if intent_decision.action == "reuse_existing":
            existing_order = session.get(PaperOrder, intent_decision.existing_order_id)  # extra per-candidate query
        ...

`derive_order_identity(prefix=..., strategy_id=..., session_date=..., symbol=..., side=..., quantity=...)`
returns an identity carrying `.intent_hash`, `.client_order_id`. It is a pure function — safe to call
per-candidate in-memory without any DB access.

`_is_resubmittable_order(order, *, failure_threshold)` and `_broker_has_touched_order(order)` are pure
predicates over an already-loaded PaperOrder row — safe to call in-memory.

`PaperOrder` has `UniqueConstraint("intent_hash")` — an exact `intent_hash` match returns at most one row.
