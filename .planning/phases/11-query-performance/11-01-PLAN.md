<objective>
Eliminate the paper-preflight N+1 query pattern so preflight (auto-resolve path) issues at most
2 SQL queries total, regardless of how many approved candidates exist.

Today `_build_paper_session_plan` (src/trading_platform/services/paper_execution.py:1184)
loops over every approved candidate and, per candidate, calls
`_resolve_paper_intent_decision` — which issues an exact-match `intent_hash` SELECT
(line 1252), a predecessor-lineage SELECT (line 1283), and sometimes a `session.get(PaperOrder)`
for reuse (line 1213). With M candidates that is ~2-3·M queries: a classic N+1. On top of that the
preflight issues one source-run-resolution query (`_resolve_source_risk_run`) and one candidate-load
query (`_load_submission_candidates`).

Purpose: Preflight cost must be flat in portfolio size (PERF-01), and the concrete bound is a literal
"at most 2 queries total" on the auto-resolve path — matching ROADMAP success criterion #1. The
assertion must be a hard, testable invariant (query count INVARIANT to candidate count AND <= 2), not
a "roughly constant" heuristic. If 2 proves genuinely infeasible after honestly attempting the fold
below, STOP and escalate to the orchestrator — do not silently ship a higher constant.
Output: A reusable query-counting test harness, a batched preflight that (a) folds source-run
resolution into the candidate query and (b) resolves all intent decisions in-memory from one PaperOrder
load, and an integration test that pins the <= 2 bound plus the non-scaling invariant.
</objective>
