Define module constants `DEFAULT_LIMIT = 20`, `MAX_LIMIT = 100`, `DEFAULT_LOG_PAGE_SIZE = 100`, `MAX_LOG_PAGE_SIZE = 500`.

Define `@dataclass(frozen=True) class JobReadFilters` with `status: str | None = None`, `job_type: str | None = None`, `limit: int = DEFAULT_LIMIT`.

Define `class JobReadService` constructed with `settings: Settings`, exposing five methods. All open their own `session_scope`, return plain JSON-serializable dicts, render datetimes with `.isoformat()`, and let no ORM object cross the service boundary — matching `operator_reads.py` exactly.

`list_jobs(filters: JobReadFilters) -> list[dict[str, Any]]` — Jobs ordered by `queued_at` descending, filtered by `status` and `job_type` when supplied, capped at `min(filters.limit, MAX_LIMIT)`. Each item carries id, job_type, status, queued_at, started_at, completed_at, failure_reason, outcome_uncertain, cancellation_requested_at, and a nested progress object with the four progress fields plus progress_updated_at.

`get_job_detail(job_id: str) -> dict[str, Any]` — raises `LookupError` naming the ID when absent. Returns the complete record: identity and job_type; all lifecycle timestamps; the terminal outcome block (failure_reason, failure_message, outcome_uncertain, result_summary); the progress snapshot; the cancellation audit block (cancellation_requested_by, cancellation_reason, cancellation_requested_at, cancellation_acknowledged_at, cancellation_cause) per D-10; the dependency causal chain (blocking_job_id, blocking_job_status, root_cause_job_id) per D-05; a `dependencies` list of every declared dependency with its ID, job_type and current status; and a `blocking_dependencies` list containing only those dependencies not yet SUCCEEDED, which is what explains to an operator why a QUEUED Job has not started.

`get_job_progress(job_id: str) -> dict[str, Any]` — raises `LookupError` when absent. Selects only the four progress columns, `progress_updated_at`, and `status` — not the whole row — because this is the endpoint a console poller hits repeatedly. Including `status` in the response means a poller learns the Job reached a terminal state from the same request rather than needing a second call.

`list_job_logs(job_id: str, *, after_sequence: int | None = None, limit: int = DEFAULT_LOG_PAGE_SIZE) -> dict[str, Any]` — raises `LookupError` when the Job does not exist; an empty list for an existing Job is a valid result, not an error. Orders strictly by `sequence` ascending (D-13 deterministic ordering — never by `logged_at`, which can collide), applies `sequence > after_sequence` when supplied, and caps at `min(limit, MAX_LOG_PAGE_SIZE)`. Returns a dict with keys `job_id`, `items`, `count`, `next_after_sequence` (the last returned sequence, or the incoming cursor when empty), and `has_more`. The sequence cursor is what makes pagination safe under concurrent appends: unlike an offset, it can neither skip nor duplicate a record while a handler is still writing. Each item carries sequence, logged_at, level, event_code, message, handler_type, context.

`list_job_events(job_id: str, *, limit: int = DEFAULT_LOG_PAGE_SIZE) -> list[dict[str, Any]]` — raises `LookupError` when absent. The lifecycle and cancellation audit trail ordered by `event_at` then `id`, capped at `min(limit, MAX_LOG_PAGE_SIZE)`, carrying from_status, to_status, event_type, outcome, event_at, requested_by, reason, requested_at, acknowledged_at, terminal_cause, and details. This is what makes JOB-06's "cancellation is audited" actually observable, and it surfaces rejected transitions too.

Module docstring: state that this is the D-15 generic read surface for JOB-07, that it is strictly read-only (no method writes), and that operation submission is Phase 18.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| untrusted HTTP client → API | Query parameters (status, job_type, limit, after_sequence) and path parameters (job_id) are attacker-controlled |
| API → database | Filter values reach SQL; unbounded limits reach the row count |
| database → HTTP response | `job_logs.context`, `jobs.payload`, and `failure_message` are returned to the caller |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-17-08-01 | Tampering | SQL injection through filter and cursor parameters | mitigate | All queries are SQLAlchemy Core/ORM constructs with bound parameters — no string-interpolated SQL. `status` is typed as the `JobStatus` enum and `job_id` as `UUID` in the route signature, so malformed values are rejected with 422 before reaching the database |
| T-17-08-02 | Denial of service | unbounded result sets on list and log endpoints | mitigate | `le=MAX_LIMIT` (100) on list, `le=MAX_LOG_PAGE_SIZE` (500) on logs and events, plus a defensive `min(limit, MAX_...)` inside the service so a non-HTTP caller cannot bypass the route bound. Test-pinned by `test_list_jobs_caps_limit` and `test_logs_reject_oversized_limit` |
| T-17-08-03 | Information disclosure | secrets surfacing through `job_logs.context` in API responses | mitigate | Context is sanitized at the single write path in plan 17-04 before persistence, so the read layer can only return already-redacted values; no unsanitized field exists to leak |
| T-17-08-04 | Information disclosure | internal error detail leaking through a 500 | mitigate | Missing entities raise `LookupError`, translated to a 404 carrying only the requested ID; malformed input is a FastAPI 422. No handler returns a raw exception |
| T-17-08-05 | Elevation of privilege | a mutating verb reaching Job state through the read router | mitigate | The router declares only GET handlers and `test_jobs_router_exposes_no_mutating_verbs` asserts the method set at runtime (D-15 scope fence) |
| T-17-08-06 | Spoofing | unauthenticated read access to Job data | accept | No auth surface ships in v1.x per REQUIREMENTS.md Out of Scope; the API is a local single-operator read surface, consistent with the existing six routers |
| T-17-08-SC | Tampering | package installs | accept | No new packages — FastAPI and SQLAlchemy are existing dependencies |
</threat_model>
