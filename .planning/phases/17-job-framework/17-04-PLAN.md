Create `src/trading_platform/jobs/context.py` defining `class DatabaseJobContext` implementing the `JobContext` Protocol.

Constructor takes `job_id: uuid.UUID`, `job_type: str`, `payload: Mapping[str, Any]`, and `settings: Settings | None = None`. It exposes `job_id`, `job_type`, and `payload` as read-only properties; `payload` returns a `MappingProxyType` so a handler cannot mutate the framework's copy.

`report_progress(*, percent=None, step=None, current=None, total=None) -> None`: builds a `ProgressSnapshot` (whose validation raises `ValueError` on a bad percent before any database access), then opens a short `session_scope`, loads the Job, calls `apply_progress`, and commits. Each call is its own transaction so progress is visible to API readers while the handler is still running — that visibility is JOB-07's "during execution" clause. Do not hold a session open across handler work.

`log(*, level, event_code, message, context=None) -> None`: this is the single Job-log write path.
  Normalize `level` to lowercase and validate it is one of debug, info, warning, error, critical — raise `ValueError` otherwise.
  Route the context dict through `sanitize()` imported from `trading_platform.core.log_sanitizer` before it is assigned to `JobLog.context`. There must be exactly one `sanitize(` call site in this module and no alternative assignment path to `JobLog.context` anywhere in the codebase (D-13). Add an inline comment stating that inventing a second sanitization path is forbidden and that `core/log_sanitizer.sanitize` is the chokepoint the LOG-06 enforcement test already pins.
  Compute `sequence` as one greater than the current maximum sequence for that `job_id`, selected inside the same transaction with a row lock on the parent Job (`session.get(Job, job_id, with_for_update=True)`) so two concurrent writers cannot collide on the `(job_id, sequence)` unique constraint. On an `IntegrityError` from a lost race, retry once, then re-raise.
  Persist `logged_at = datetime.now(UTC)`, `level`, `event_code`, `message`, `handler_type = self.job_type`, `job_id`, `context = sanitized`.
  Enforce a log-volume safeguard: truncate `message` to 4000 characters and reject a `context` dict whose serialized JSON exceeds 16384 bytes by replacing it with `{"_truncated": True, "_original_bytes": <n>}`. Define both limits as module constants `MAX_LOG_MESSAGE_CHARS = 4000` and `MAX_LOG_CONTEXT_BYTES = 16384` so the API layer in plan 17-08 can reference them.

`is_cancellation_requested() -> bool`: opens a short read-only `session_scope`, reads `Job.cancellation_requested_at`, returns True when it is not None. Cache the True result in an instance flag so repeated safe-point checks in a tight loop do not hammer the database; never cache a False result.

`raise_if_cancelled() -> None`: calls `is_cancellation_requested()` and raises `JobCancelledError(job_id=self.job_id)` when True.

Add a module docstring stating: this class is the only object a handler receives; it exposes no session, no engine, and no status setter, which is what keeps JOB-04's boundary true at runtime rather than only statically.
  </action>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| handler-supplied log context → `job_logs.context` | Handler code passes arbitrary dicts that may contain credentials, tokens, connection URLs, or broker order IDs |
| handler-supplied message/context volume → database | An unbounded or looping handler can write unlimited log volume |
| framework → handler | The context object is the entire capability set granted to handler code |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-17-04-01 | Information disclosure | `JobLog.context` | mitigate | Every context dict passes through `trading_platform.core.log_sanitizer.sanitize` at the single write site in `context.py`; enforced by an exact `grep -c "sanitize("` count of 1 and by `test_log_context_is_sanitized` |
| T-17-04-02 | Denial of service | unbounded log message/context size | mitigate | `MAX_LOG_MESSAGE_CHARS = 4000` truncation and `MAX_LOG_CONTEXT_BYTES = 16384` context replacement, test-pinned |
| T-17-04-03 | Denial of service | unbounded log record count per Job | accept | D-14 forbids pruning in Phase 17; per-record size caps bound the worst case and the read-side page cap lands in plan 17-08 |
| T-17-04-04 | Elevation of privilege | handler obtaining a database session or status setter through the context | mitigate | `DatabaseJobContext`'s public surface is asserted equal to the seven Protocol members; no engine/session-factory import is permitted (grep-enforced) |
| T-17-04-05 | Tampering | concurrent writers colliding on `(job_id, sequence)` | mitigate | Sequence assignment happens under a `with_for_update` row lock on the parent Job, with a single retry on `IntegrityError` |
| T-17-04-SC | Tampering | package installs | accept | No new packages |
</threat_model>
