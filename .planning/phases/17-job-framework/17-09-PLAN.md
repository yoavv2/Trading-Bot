Create `src/trading_platform/jobs/runner.py` with `execute_job(*, job_id: uuid.UUID, worker_id: str, registry: JobRegistry, settings=None) -> JobStatus`.

Read the Job's `job_type` and `payload` in a short `session_scope`, then close it. Resolve the handler via `registry.resolve(job_type)`. An `UnknownJobTypeError` means a Job exists for a type this worker cannot run: transition to FAILED with `failure_reason=JobFailureReason.HANDLER_ERROR`, a `failure_message` naming the unknown type, and `outcome_uncertain=False` — nothing ran, so nothing is uncertain. Cascade to dependents and return.

Construct a `DatabaseJobContext(job_id=..., job_type=..., payload=..., settings=...)`.

Start a heartbeat: a daemon thread driven by a `threading.Event` wait loop that calls `renew_lease` every `HEARTBEAT_SECONDS`. When `renew_lease` returns False the lease was reclaimed by a sweep — set a `lease_lost` flag and stop the thread. Stop and join the thread in a `finally` block on every exit path so no thread outlives the Job.

Call `handler.run(context)` with no database session open. This mirrors the DB-04/DB-05 convention already enforced in `paper_execution.py`: external side effects must never run inside a transaction.

Land the outcome, each in its own `session_scope`. Before writing any terminal transition, check `lease_lost`; if set, log a WARNING through `get_logger` and return without transitioning — a worker must not write a terminal state for a Job another sweep already reclaimed.
  Normal return: call `mark_completed` (progress to 100 — D-12's only guaranteed-100 path), then `apply_job_transition` with `event_type=JobEventType.SUCCEEDED` and `result_summary` set to the handler's returned mapping.
  `JobCancelledError`: call `acknowledge_cancellation` (D-08). Do not call `apply_job_transition` directly here — acknowledgement owns that transition and carries the D-10 audit fields forward.
  Any other exception: `apply_job_transition` with `event_type=JobEventType.FAILED`, `failure_reason=JobFailureReason.HANDLER_ERROR`, and `failure_message` set to `f"{type(exc).__name__}: {exc}"` truncated to 2000 characters. Set `outcome_uncertain=True` when the Job emitted at least one `job_logs` row whose `event_code` starts with `external_` — the framework cannot prove whether an external side effect occurred (D-03); otherwise False. Do not put the traceback in `failure_message`; emit it through `get_logger(...).exception(...)` so the sanitizing `JsonLogFormatter` handles it. Progress is left untouched (D-12).
  On any FAILED or CANCELLED outcome, call `cascade_dependency_outcome` in the same transaction (D-04).

Return the resulting `JobStatus`.

Module docstring: state that this module contains no job-type literals and no domain behavior. It resolves handlers by a string key read from the database, which is precisely what makes JOB-03's "adding a Job type touches zero queue-framework modules" true.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| handler code → worker process | Handler code runs in-process and may raise, hang, or produce external side effects |
| handler exception text → persisted failure_message | Exception strings can carry credentials or connection URLs |
| worker process → Job terminal state | A worker whose lease was reclaimed could overwrite a already-settled outcome |
| CLI arguments → worker process | `--worker-id` and `--max-jobs` are operator-supplied |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-17-09-01 | Information disclosure | credentials leaking through `failure_message` or a traceback | mitigate | `failure_message` is the truncated `type: message` string only; the full traceback is emitted through `get_logger(...).exception(...)`, whose `JsonLogFormatter` sanitizes the whole assembled payload |
| T-17-09-02 | Elevation of privilege | a worker writing terminal state for a Job it no longer owns | mitigate | `renew_lease` returning False sets a `lease_lost` flag checked before any terminal transition; pinned by `test_worker_that_lost_its_lease_writes_no_terminal_state` |
| T-17-09-03 | Spoofing | reporting SUCCEEDED for a handler whose external side effect state is unknown | mitigate | D-03: an `external_*` log event before a failure forces `outcome_uncertain=True`; SUCCEEDED is written only on a clean handler return |
| T-17-09-04 | Denial of service | a hung handler holding a lease indefinitely | mitigate | The cancellation grace-period sweep (plan 17-06) runs on every poll and lands an unresponsive Job on FAILED / `cancellation_timeout`; the heartbeat stops when the process dies |
| T-17-09-05 | Denial of service | orphaned leases on worker shutdown | mitigate | SIGTERM/SIGINT handlers set a shutdown event so the in-flight Job finishes and the heartbeat thread is joined in a `finally` block |
| T-17-09-06 | Tampering | operator-supplied `--worker-id` colliding with another worker | accept | Worker IDs are advisory labels for audit, not authorization; claim exclusivity rests on `SKIP LOCKED` row locks, not on ID uniqueness. Default is host:pid, which is collision-resistant in practice |
| T-17-09-SC | Tampering | package installs | accept | No new packages — `threading`, `signal`, `socket`, `os` are stdlib |
</threat_model>
