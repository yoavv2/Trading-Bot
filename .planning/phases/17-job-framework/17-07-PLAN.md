`claim_next_job(session, *, worker_id: str, now: datetime | None = None, lease_seconds: int = LEASE_SECONDS) -> uuid.UUID | None`:
  Build `select(Job.id)` filtered on `Job.status == JobStatus.QUEUED`, `Job.cancellation_requested_at.is_(None)`, and `~unsatisfied_dependency_exists(Job.id)` imported from `dependencies.py` — readiness has exactly one definition in the codebase and this reuses it rather than restating the SQL. Order by `Job.queued_at`, then `.limit(1).with_for_update(skip_locked=True)`.
  `SKIP LOCKED` is the mechanism that makes concurrent workers never double-claim: a row another worker's open transaction has locked is skipped rather than waited on, so the second worker gets the next eligible Job or nothing. Add an inline comment stating that.
  The `cancellation_requested_at IS NULL` predicate inside the same locked statement is what guarantees D-07: a Job cancelled while queued can never be claimed, so its handler is never invoked.
  On a hit: set `lease_owner = worker_id`, `lease_expires_at = now + timedelta(seconds=lease_seconds)`, `heartbeat_at = now`, then call `apply_job_transition` with `event_type=JobEventType.CLAIMED`, which sets status RUNNING and `started_at`. Return the Job ID. Return None when nothing is eligible. Never commit — the caller owns the transaction.

`renew_lease(*, job_id, worker_id, lease_seconds=LEASE_SECONDS, settings=None) -> bool`: in its own short `session_scope`, extend `lease_expires_at` and set `heartbeat_at` only when `status == JobStatus.RUNNING` and `lease_owner == worker_id`. Return False otherwise. A False return means the lease was already reclaimed by a sweep, and the runner in plan 17-09 uses it to stop working on a Job it no longer owns.

`find_lost_job_ids(session, *, now=None) -> list[uuid.UUID]`: a single `select` mirroring `find_stale_runs`: `Job.status == JobStatus.RUNNING` and `Job.lease_expires_at < now`. No separate liveness protocol is needed — a dead process cannot heartbeat, so its lease simply lapses.

`reclaim_lost_jobs(session, *, now=None, failure_reason: JobFailureReason = JobFailureReason.LEASE_EXPIRED) -> list[uuid.UUID]`: for each lost Job, call `apply_job_transition` with `event_type=JobEventType.LEASE_EXPIRED`, the given `failure_reason`, and a `failure_message` naming the previous lease owner and the expiry timestamp. Per D-01 the result is FAILED, never CANCELLED; per D-03 `outcome_uncertain` is forced True by the lifecycle module, so this function does not need to pass it. Clear `lease_owner` and `lease_expires_at`. After each terminal transition call `cascade_dependency_outcome` so descendants are not stranded (D-04). Return the reclaimed IDs. Flush, never commit.
  Idempotency mirrors `reclaim_stale_runs` exactly: once a Job leaves RUNNING it no longer matches the detector predicate, so a second pass returns an empty list. Do not add a reclaimed flag.
  Per D-02, add an inline comment: this function must never set a Job back to QUEUED. Phase 17 has no automatic requeue or retry — recovery is deterministic and visible, and explicit retry is Phase 19 (OPS-07).

Module docstring: state the JOB-02 guarantee and name the two mechanisms (`SKIP LOCKED` against duplication, lease expiry against loss).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| worker process ↔ Job queue | Multiple worker processes contend for the same rows; a crashed worker leaves claimed state behind |
| cancel path ↔ claim path | A QUEUED Job could be cancelled and claimed concurrently, invoking a handler for a cancelled Job |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-17-07-01 | Tampering | two workers executing the same Job | mitigate | `SELECT ... FOR UPDATE SKIP LOCKED` on the claim query plus the absorbing QUEUED→RUNNING transition; proven by `test_concurrent_workers_never_claim_the_same_job` with two real connections |
| T-17-07-02 | Denial of service | Jobs lost forever when a worker crashes | mitigate | Lease-expiry detector plus `reclaim_lost_jobs`; a dead process cannot heartbeat so the lease lapses without any liveness protocol |
| T-17-07-03 | Spoofing | reporting a definite outcome for a Job whose side effect is unknown | mitigate | D-03: reclaimed Jobs are FAILED with `outcome_uncertain=True`, forced by the lifecycle module rather than left to this caller |
| T-17-07-04 | Elevation of privilege | invoking a handler for a Job already cancelled while queued | mitigate | `cancellation_requested_at IS NULL` is a predicate of the same row-locked claim statement (D-07) |
| T-17-07-05 | Denial of service | dependents stranded behind a crashed ancestor | mitigate | `reclaim_lost_jobs` calls `cascade_dependency_outcome` on every terminal transition (D-04) |
| T-17-07-SC | Tampering | package installs | accept | No new packages |
</threat_model>
