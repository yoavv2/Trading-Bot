<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| handler code → framework | Third-party-shaped handler code runs inside the worker process and can only affect Job state through `JobContext` |
| domain services → framework | Architectural boundary: services must remain callable without the Job framework present |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-17-02-01 | Elevation of privilege | Job handler writing directly to `jobs` table | mitigate | `JobHandler.run` receives only a `JobContext`; the contract docstring forbids lifecycle writes and the context exposes no session or status setter |
| T-17-02-02 | Tampering | duplicate/shadowing job type registration | mitigate | `JobRegistry.register` raises `ValueError` on duplicate keys; no silent overwrite path exists |
| T-17-02-03 | Elevation of privilege | domain service reaching into HTTP/scheduling layers | mitigate | `tests/test_job_import_boundary.py` AST scan over all service modules with a non-emptiable scope guard (JOB-04) |
| T-17-02-04 | Information disclosure | handler-supplied log context | accept (deferred to 17-04) | The contract declares `log(context=...)`; sanitization is implemented at the single write path in plan 17-04 |
| T-17-02-SC | Tampering | package installs | accept | No new packages — `ast` and `pytest` are stdlib/existing dependencies |
</threat_model>
