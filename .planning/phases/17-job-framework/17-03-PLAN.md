  QUEUED + CANCELLED → CANCELLED (D-07 — atomic queued cancel, handler never invoked);

SUCCEEDED, FAILED, and CANCELLED map to empty dicts — terminal states have zero outgoing edges. Add an inline comment stating that `CANCELLATION_REQUESTED` and `DEPENDENCY_RESOLVED` are deliberately absent: they are recorded as JobEvent rows by plans 17-06 and 17-05 without changing status, so they must not appear in this table.

Define `class IllegalJobTransition(RuntimeError)` taking keyword args `job_id`, `from_status`, `event_type` and a `details` mapping, with a message of the form `f"Illegal job transition for {job_id}: {from_status.value} -> {event_type.value}"`, mirroring `IllegalOrderTransition`.

Define a frozen dataclass `JobTransitionRequest` with fields: `event_type: JobEventType`, `event_at: datetime | None = None` (defaults to `datetime.now(UTC)`), `failure_reason: JobFailureReason | None = None`, `failure_message: str | None = None`, `outcome_uncertain: bool | None = None`, `cancellation_cause: JobCancellationCause | None = None`, `requested_by: str | None = None`, `reason: str | None = None`, `requested_at: datetime | None = None`, `acknowledged_at: datetime | None = None`, `result_summary: Mapping[str, Any] | None = None`, `blocking_job_id: uuid.UUID | None = None`, `blocking_job_status: JobStatus | None = None`, `root_cause_job_id: uuid.UUID | None = None`, `details: Mapping[str, Any] = field(default_factory=dict)`.

Define a frozen dataclass `JobTransitionResult` with `job_id`, `from_status`, `to_status`, `event_id`, `outcome`.

Define `apply_job_transition(session: Session, *, job_id: uuid.UUID, request: JobTransitionRequest) -> JobTransitionResult`. It takes an open `Session` (the caller owns the transaction boundary, matching `reclaim_stale_runs`'s convention — it flushes, never commits) and:
  1. Loads the Job with `session.get(Job, job_id, with_for_update=True)` so concurrent workers serialize on the row; raise `LookupError` if absent.
  2. Resolves the target status. If None, appends a JobEvent with `outcome=JobTransitionOutcome.REJECTED`, `from_status=<current>`, `to_status=None`, the request's event_type, and `details` including the attempted transition, flushes, then raises `IllegalJobTransition`. The rejected row must be persisted, not discarded — this is the audit half of JOB-01.
  3. On a legal transition: sets `job.status`, sets `job.started_at = event_at` when the target is RUNNING (only if currently None), sets `job.completed_at = event_at` when the target is terminal.
  4. Applies the terminal-outcome fields when present: `failure_reason`, `failure_message`, `cancellation_cause`, `result_summary`, `blocking_job_id`, `blocking_job_status`, `root_cause_job_id`. `outcome_uncertain` is set only when the request supplies it explicitly, except that WORKER_LOST, LEASE_EXPIRED, and CANCELLATION_TIMEOUT force `outcome_uncertain=True` regardless (D-03, D-09) — implement that as an explicit rule, not as a caller obligation.
  5. Enforces D-01 defensively: if `event_type` is WORKER_LOST or LEASE_EXPIRED and no `failure_reason` was supplied, default it to the matching `JobFailureReason` member; if the target status is CANCELLED and a `failure_reason` was supplied, raise `ValueError` — infrastructure failure must never be recorded as a cancellation.
  6. Enforces D-12: this function never writes to `progress_percent`, `progress_step`, `progress_current`, `progress_total`, or `progress_updated_at`. Add an inline comment saying so.
  7. Appends one `JobEvent` with `outcome=ACCEPTED`, from/to statuses, event_at, the D-10 fields (`requested_by`, `reason`, `requested_at`, `acknowledged_at`), `terminal_cause` set to the `failure_reason.value` or `cancellation_cause.value` when either is present, and `details`.
  8. Calls `session.flush()` and returns the `JobTransitionResult`.

Add a module docstring stating this is the only authorized writer of `Job.status`, that terminal states are absorbing, and that callers own the transaction boundary.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| concurrent workers → same Job row | Two workers may attempt to transition the same Job simultaneously |
| caller-supplied transition request → persisted terminal outcome | Callers could otherwise record an infrastructure failure as an operator cancellation |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-17-03-01 | Tampering | concurrent double transition of one Job | mitigate | `session.get(Job, job_id, with_for_update=True)` takes a row lock; terminal states are absorbing so a second transition raises rather than overwriting |
| T-17-03-02 | Repudiation | a status change with no audit trail | mitigate | Every accepted and every rejected transition writes a `JobEvent` in the same transaction as the status update; no code path updates status without one |
| T-17-03-03 | Spoofing | infrastructure failure recorded as operator cancellation | mitigate | `apply_job_transition` raises `ValueError` when a `failure_reason` accompanies a CANCELLED target; WORKER_LOST/LEASE_EXPIRED/CANCELLATION_TIMEOUT force FAILED plus `outcome_uncertain=True` (D-01, D-03, D-09) |
| T-17-03-04 | Information disclosure | `JobEvent.details` and `failure_message` | mitigate | Callers pass only framework-generated transition metadata; handler-supplied text reaches the database only through the sanitized log path in plan 17-04 |
| T-17-03-SC | Tampering | package installs | accept | No new packages |
</threat_model>
