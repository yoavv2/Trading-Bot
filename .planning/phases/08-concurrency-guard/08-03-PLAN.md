<objective>
Provide stale-run detection and lazy reclaim: a single-query detector for `running` runs past the configured timeout (LOCK-04), and a tuple-scoped reclaim that marks ALL past-threshold rows `STALE` and records a durable `ExecutionEvent` (LOCK-05), reusing Phase 7's `StrategyRun` + `ExecutionEvent` audit pattern rather than inventing a parallel channel.

Purpose: When a new run wins the lock but finds a leftover `running` row from a crashed predecessor, it must explicitly reclaim it — visible in the audit trail — then continue.
Output: `services/stale_runs.py` + a Postgres integration test.
</objective>
