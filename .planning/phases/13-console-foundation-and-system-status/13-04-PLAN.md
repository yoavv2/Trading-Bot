<!-- Endpoint payloads (verified against src/trading_platform/api/routes/): -->

GET /health → 200:
`{"status":"ok","service":<slug>,"version":<v>,"timestamp":<iso>}`

GET /ready → 200 (ready) OR 503 (degraded/starting) — BOTH carry the same JSON body:
`{"status":"ready"|"degraded"|"starting","ready":bool,"started_at":...,"timestamp":...,
  "checks":{"application":{"status","detail"},"configuration":{...},
            "database":{"status":"ok"|"error"|"skipped","detail":...,"required":bool}}}`
NOTE: a 503 here is DATA, not just an error — fetchApi surfaces the parsed body in `failure.body`,
and ReadinessPanel must render the degraded checks from it (this is the honest-render case).

GET /api/v1/system → 200:
`{"application":{"name","version","environment","operator_mode"},
  "api":{...},"strategy_catalog":[...],
  "database":{"driver","host","port","name","readiness_checks_enabled","readiness_required","schema_managed_by"},
  "operator_read_api":{...}}`

GET /api/v1/runs?limit=1 → 200 (defaults: strategy_id=trend_following_daily, no run_type/status filter,
ordered started_at DESC — so items[0] IS the latest run of any type):
`{"filters":{...},"count":N,"items":[{"run_id","strategy_id","display_name","run_type","status",
  "trigger_source","as_of_session","started_at","completed_at","parameters_snapshot",
  "result_summary","error_message"}]}`

GET /api/v1/system/kill-switch → 200 (from plan 13-01):
`{"name","state":"armed"|"tripped","is_tripped":bool,"last_changed_at","last_change_actor",
  "last_change_reason","last_change_run_id"}`

Shared contracts from plan 13-03: `fetchApi<T>`, `useApiQuery<T>` → `{loading, result, refetch}`,
`<ErrorState failure=... />`, `<FetchMeta asOf loading onRefresh />`.
</interfaces>
