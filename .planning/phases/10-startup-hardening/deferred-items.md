# Deferred Items — Phase 10 (Startup Hardening)

## From 10-02 (Log Sanitization Core)

**`emit_structured_log`'s chokepoint covers the `context` dict only, not the `message` positional argument.**

`sanitize()` is applied to the assembled `context` dict before `logger.log(...)`, satisfying this plan's declared scope (Task 2's action text: "the `context` dict is `sanitize(context, unmask_ids=...)` before `logger.log`"). However, the `message` string passed to `emit_structured_log`/`logger.log` is NOT scrubbed — `JsonLogFormatter.format()` emits `record.getMessage()` raw. A call like `emit_structured_log(logger, INFO, "connecting with password=hunter2")` would leak the secret in the `message` field of the emitted JSON line, even though any secret placed in `context` is redacted.

This is intentional and in-scope-as-written: this plan's `<verification>` section explicitly scopes LOG-06 (the emitted-line enforcement test, which would need to cover full-payload including `message`) to plan 10-06, and the plan's `key_facts` names "the formatter as a defense-in-depth backstop" as the mechanism that would close this gap — also 10-06's territory (`10-06-PLAN.md — Logger migration + formatter backstop + import-boundary & emitted-line enforcement tests`).

**Action for 10-06:** When adding the formatter backstop, sanitize `record.getMessage()` (or the raw `message` string before formatting) in addition to `context`, so LOG-06's "no emitted log line contains `password=`..." enforcement test can assert over the whole JSON line, not just the `context` sub-object. Do not assume today's "single sanitization chokepoint" claim (10-02-SUMMARY.md, `get_logger` docstring) covers the message string — it covers `context` only.

**Minor, non-blocking:** `configure_logging()`'s copy of `settings.debug_unmask_ids` into the module-level `_DEBUG_UNMASK_IDS` flag has no dedicated test (the existing unmask test sets the module global directly). Trivial one-line assignment, low risk; worth a quick assertion if 10-06 touches `configure_logging` anyway.

## From 10-05 (Startup Hardening Wiring)

**Pre-existing, out-of-scope flake: intermittent `psycopg.errors.InsufficientPrivilege: must be a superuser to terminate superuser process` at teardown of `migrated_access_db`-style fixtures** (`tests/test_market_data_access.py`, `tests/test_market_data_ingestion.py`, `tests/test_portfolio_service.py`, `tests/test_risk_pipeline.py`, and others sharing the same per-test-DB-provisioning pattern). Each of these fixtures creates a scratch database, runs migrations, yields, then on teardown calls `pg_terminate_backend` against every backend connected to that database before `DROP DATABASE`. When many such fixtures run back-to-back across the full suite, teardown occasionally races a backend that is itself a superuser session (or one that closed between the `pg_stat_activity` read and the `pg_terminate_backend` call), which Postgres refuses with `InsufficientPrivilege`.

**Confirmed pre-existing (not caused by 10-05), with reproduction evidence, not just inference:** checked out the commit immediately before this plan's first commit (`24e58c3`, before 10-04 and 10-05 both) into an isolated `git worktree` and ran `python -m pytest -q` three times against the identical, untouched pre-10-05 code — the same flake reproduced every time (1-3 different, non-deterministic test files per run: `test_market_data_access.py` (multiple distinct tests across runs), `test_risk_pipeline.py`). This rules out both 10-04's and 10-05's changes as the cause. Independent supporting evidence: the failing `pg_terminate_backend` targets a *superuser* backend, but every fixture in this codebase (including this plan's `enforce_startup_config` DB-reachability check) connects as the configured non-superuser `trading_platform` DB user — the gate's own connections cannot be the session that fails to terminate. The most plausible origin is 10-03 (this same phase, landed earlier), which rewrote the engine/session-cache lifecycle in `db/session.py` that every one of these fixtures' `session_scope`/`get_engine` calls goes through — a connection-pool/cache-keying change is a far more likely source of a teardown-time backend-visibility race than this plan's startup-gate wiring, though this plan did not further isolate 10-03 specifically as the cause (out of scope; noted for whoever picks this up).

Out of this plan's `files_modified` scope (test infra shared across ~6+ unrelated test files). A follow-up should either serialize/guard the `pg_terminate_backend` teardown step (e.g. catch and ignore `InsufficientPrivilege`, or filter `usename = current_user` as `test_concurrency_guard_e2e.py`'s own fixture already does) or investigate why some backend in `pg_stat_activity` is unexpectedly a superuser session during these tests — starting from 10-03's engine-cache change as the leading hypothesis.
