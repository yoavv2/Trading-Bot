<objective>
Build the pure, entrypoint-agnostic config validation core: a closed ExecutionMode enum, a typed ConfigValidationError carrying the failed field name and expected shape, and validate_config(payload, *, mode) -> Settings that validates the RAW pre-pydantic config payload and returns a constructed Settings (or raises a single aggregated, actionable error). Validating the raw payload — not an already-constructed Settings — is what makes CFG-05 (tolerance bounds) actually reachable: pydantic's Field(ge/le) bounds otherwise reject out-of-range values with a raw ValidationError at Settings.model_validate() time, before any semantic validator could name the field.

This plan produces NO wiring into entrypoints and NO process-exit behavior — that is plan 10-05. This plan is the contract 10-05 consumes.

Purpose: Startup must refuse invalid config with a message naming the failed field, but the decision of what is valid must be a pure, unit-testable function decoupled from where the process boots.
Output: core/config_validation.py + full unit test suite.
</objective>

<key_facts>
- There is NO mode field anywhere in Settings (verified). Execution mode is implicit in the CLI subcommand / entrypoint. DECISION for this phase: mode is passed as an explicit parameter validate_config(payload, *, mode: ExecutionMode) by each entrypoint (10-05 wires it) — a config mode field is deliberately NOT added, because a field could contradict the running command.
- CRITICAL for CFG-05: the tolerance fields already carry pydantic Field(ge/le) bounds (TrendFollowingRiskSettings.risk_per_trade ge=0/le=1; ExecutionSafetySettings.repeated_failure_threshold ge=1; ExecutionSafetySettings.stale_run_timeout_minutes ge=1; BacktestSlippageSettings.bps ge=0; PortfolioSettings.max_strategy_allocation_pct gt=0/le=1). Because pydantic enforces these at Settings.model_validate(), a range check on an already-constructed Settings is DEAD CODE — the bad value is rejected earlier as a raw pydantic ValidationError. Therefore validate_config must operate on the RAW payload and OWN the Settings.model_validate call, translating any pydantic ValidationError into a typed, field-named ConfigValidationError (loc -> field path). The "declared typed bounds" CFG-05 references ARE the pydantic Field bounds — do NOT duplicate them in a hand-maintained min/max table (drift risk); reuse pydantic and translate its error.
- Raw payload source: build_settings_payload(...) in settings.py returns the merged dict BEFORE Settings.model_validate. 10-05 will call validate_config(build_settings_payload(), mode=...).
- There is no live-broker config today. The LIVE enum member exists per the closed-set spec, but its cross-field branch guards config that does not yet exist (e.g. asserts no live endpoint is configured while mode=paper). Write it to the spec; do not invent a live broker settings block.
- broker.provider is Literal["alpaca"]; broker.alpaca.api_key/.api_secret default ""; .base_url defaults to the paper endpoint. market_data.polygon.api_key defaults "".
