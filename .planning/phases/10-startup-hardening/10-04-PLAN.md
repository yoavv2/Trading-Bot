<objective>
Make the paper-order submission flow's transaction semantics explicit and safe: every state-affecting write runs inside an explicit transaction boundary (DB-04), the commit that records broker success happens only after BOTH the broker call succeeded AND the state transition persisted (DB-05), and when a rollback occurs after the broker has already accepted an order, a reconciliation is scheduled rather than the failure being silently swallowed (DB-06).

Scoped strictly to `paper_execution.py` (the only flow with a broker side effect). `session_scope` is already the explicit boundary and the broker call is already made outside the persisting session — so DB-04/05 are mostly about making the invariant explicit and test-pinned; DB-06 is the genuinely new behavior.

Purpose: A broker order that succeeds but whose local persistence rolls back leaves the DB and broker divergent — reconciliation (Phase 9) exists precisely to detect this, so the partial-failure path must schedule it.
Output: hardened submission transaction boundaries + reconciliation scheduling + tests.
</objective>

<key_facts>
- Verified submission structure in `paper_execution.py`: (1) pre-broker session_scope (~lines 439-539) registers the PENDING_SUBMISSION order + intent transition and commits BEFORE the broker call — this is deliberate idempotency (intent durable before submit). (2) The broker call `broker_execution.submit_order(intent)` (~line 561) is made OUTSIDE any open session. (3) On broker exception (~562-581) a NEW session_scope records SUBMISSION_FAILED and re-raises. (4) On broker success, a NEW session_scope (~583-609) persists `broker_order_id`/`broker_status`/`submitted_at` and applies the success transition, committing at scope exit. So the broker call is already outside the transaction and the success-commit already follows the broker result — DB-05's ordering is structurally present.
- DB-06 gap: if the post-broker persist session (~583-609) raises and rolls back, the broker already accepted the order but local state never records it. Current code just re-raises — no reconciliation is scheduled. This is the DB-06 scenario to fix.
- The reconciliation entrypoint exists: `reconcile_paper_execution` (read-only) and `apply_reconciliation_corrections` in `services/reconciliation.py`; the session runner already runs recover→reconcile→correction after submission. There is no async task queue — "schedule/enqueue" here means a durable, testable hand-off (invoke/enqueue a reconciliation marker) rather than a background worker.
- `session_scope` (db/session.py) commits on clean exit and rolls back on exception — it IS the "with session.begin() or equivalent" DB-04 boundary.
</key_facts>
